## 📖 О проекте

**Farmika** — это AI-ассистент, разработанный для помощи сотрудникам аптечных сетей в решении повседневных рабочих вопросов. Система работает полностью на **локальной инфраструктуре**, без отправки данных во внешние облачные сервисы, что критически важно для фармацевтической отрасли с её строгими требованиями к конфиденциальности.

### Какие задачи решает

- **Техническая поддержка** — инструкции по устранению неполадок с банковскими терминалами, кассовым оборудованием и программным обеспечением
- **Регламентные вопросы** — порядок действий при ошибках обмена с МДЛП (Маркировка лекарственных препаратов), работа с кассовыми программами
- **Распознавание документов** — OCR-модуль для извлечения текста из фотографий документов, рецептов и этикеток
- **Обучение новых сотрудников** — быстрый доступ к регламентам и пошаговым инструкциям без необходимости обращения к руководству

### Ключевые особенности

- 🔒 **Полная приватность** — все данные обрабатываются локально, никакие запросы не покидают внутреннюю сеть
- 🧠 **Русскоязычная LLM** — базовая модель [Vikhr-Nemo-12B-Instruct](https://huggingface.co/Vikhrmodels/Vikhr-Nemo-12B-Instruct-R-21-09-24), специализированная на русском языке
- ⚡ **8-bit квантизация** — оптимизация потребления видеопамяти через `bitsandbytes` для работы на потребительских GPU
- 🎯 **LoRA fine-tuning** — дообучение на реальных данных из аптечных регламентов без изменения весов базовой модели
- 📸 **OCR-модуль** — встроенное распознавание текста с изображений через EasyOCR (русский + английский)
- 📊 **Система обратной связи** — сбор фидбека от пользователей для итеративного улучшения качества ответов
- 🛡️ **Антигаллюцинации** — пост-обработка ответов модели с обрезкой типичных галлюцинаций и ограничением длины

## 🧠 Модель

### Базовая модель

Проект использует **[Vikhr-Nemo-12B-Instruct-R-21-09-24](https://huggingface.co/Vikhrmodels/Vikhr-Nemo-12B-Instruct-R-21-09-24)** — русскоязычную инструктивную модель на 12 миллиардов параметров от Vikhr Models. Модель выбрана за высокое качество генерации на русском языке и совместимость с форматом инструкций `[INST]`.

### Квантизация

При инференсе модель загружается в **8-bit** режиме через `BitsAndBytesConfig`, что позволяет уместить 12B модель в ~12 ГБ видеопамяти вместо ~24 ГБ при полной точности.

При дообучении используется ещё более агрессивная **4-bit NF4 квантизация** с двойным квантованием, что позволяет тренировать LoRA-адаптеры на GPU с 10–12 ГБ VRAM.

### LoRA-адаптеры

Дообучение выполняется методом **QLoRA** (Quantized Low-Rank Adaptation):

| Параметр | Значение | Пояснение |
|----------|----------|-----------|
| Rank (r) | 16 | Баланс между качеством и потреблением памяти |
| Alpha | 32 | Масштабирующий коэффициент LoRA |
| Target modules | q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj | Все линейные слои трансформера |
| Dropout | 0.0 | Не используется — минимальное влияние на качество |
| Task type | CAUSAL_LM | Авторегрессионная генерация |

### Пост-обработка ответов

Для борьбы с галлюцинациями модели применяются два механизма:

1. **Обрезка по стоп-фразам** — ответ обрезается при обнаружении типичных «хвостов» галлюцинаций: `"Если не помогло — обратитесь"`, `"Также вы можете"`, `"В заключение"` и других
2. **Ограничение длины** — ответ ограничивается 3 предложениями для лаконичности и точности

### Параметры генерации

| Параметр | Значение | Назначение |
|----------|----------|------------|
| max_new_tokens | 200 | Максимальная длина генерации |
| temperature | 0.95 | Умеренная случайность для разнообразия |
| top_p | 0.9 | Nucleus sampling |
| repetition_penalty | 1.2 | Штраф за повторения |

## 📋 Требования

### Аппаратные

- **GPU**: NVIDIA с поддержкой CUDA и минимум 10 ГБ VRAM (рекомендуется RTX 3080/3090 или выше)
- **RAM**: минимум 16 ГБ оперативной памяти
- **Диск**: ~25 ГБ свободного места для весов модели

### Программные

- Python 3.10+
- CUDA Toolkit 11.8+ (совместимый с вашей версией PyTorch)
- Conda (рекомендуется) или virtualenv

## 🚀 Установка

### 1. Клонирование репозитория

```bash
git clone https://github.com/araizze/farmika.git
cd farmika
```

### 2. Создание виртуального окружения

```bash
# Вариант с conda (рекомендуется)
conda create -n farmika python=3.10
conda activate farmika

# Или с venv
python -m venv .venv
source .venv/bin/activate  # Linux/Mac
.venv\Scripts\activate     # Windows
```

### 3. Установка зависимостей

```bash
pip install -r requirements.txt
```

#### Содержание `requirements.txt`:

| Пакет | Назначение |
|-------|------------|
| `fastapi` | Web-фреймворк для API |
| `uvicorn` | ASGI-сервер для FastAPI |
| `transformers` | Загрузка и инференс HuggingFace-моделей |
| `torch` | Фреймворк глубокого обучения |
| `accelerate` | Оптимизация загрузки модели и `device_map="auto"` |

| `peft` | Библиотека для LoRA/QLoRA |
| `easyocr` | Распознавание текста на изображениях |

### 4. Инициализация базы данных (для админ-панели)

```bash
python create_tables.py
```

Это создаст SQLite-базу `db.sqlite3` с таблицами `dataset` и `feedback`.

## ▶️ Запуск

### Основной сервер (инференс)

```bash
python -m uvicorn server:app --reload --port 8000
```

При первом запуске модель будет скачана с HuggingFace (~24 ГБ) и кэширована локально. Последующие запуски используют кэш.

В логах вы увидите:

```
🟢 Python running: /path/to/python
True                              # CUDA доступна
Ищем LoRA-модель в папке: vikhr_nemo_12B_lora_out
✅ LoRA-модель успешно загружена!  # Если адаптеры обучены
📦 Модель загружена на устройство: cuda:0
```

### Админ-панель

```bash
python -m uvicorn admin_app:app --reload --port 8001
```

Админ-панель доступна по адресу `http://localhost:8001/admin`.

## 📡 API Endpoints

### `GET /health`

Проверка состояния сервера.

```bash
curl http://localhost:8000/health
```

```json
{"status": "ok"}
```

### `POST /ask`

Основной эндпоинт — задать вопрос AI-консультанту.

**Запрос:**

```bash
curl -X POST http://localhost:8000/ask \
  -H "Content-Type: application/json" \
  -d '{"prompt": "Проблема со старым банковским терминалом"}'
```

**Ответ:**

```json
{
  "response": "При проблемах со старым банковским терминалом необходимо выполнить следующие действия: выйти из кассовой программы, выключить кассовый компьютер, обязательно выключить бесперебойник, чтобы терминал перезагрузился, проверить, что дисплей банковского терминала погас, затем включить все заново и пробить чек."
}
```

### `POST /ocr`

Распознавание текста с изображения (multipart/form-data).

```bash
curl -X POST http://localhost:8000/ocr \
  -F "file=@photo.jpg"
```

**Ответ (успех):**

```json
{
  "text": "Распознанный текст с изображения",
  "message": "✅ Распознавание завершено успешно."
}
```

**Ответ (текст не найден):**

```json
{
  "text": "",
  "message": "❗️Текст на изображении не найден."
}
```

### `POST /feedback`

Отправка обратной связи о качестве ответа.

```bash
curl -X POST http://localhost:8000/feedback \
  -H "Content-Type: application/json" \
  -d '{
    "user_id": 12345,
    "prompt": "При обмене с МДЛП ошибка",
    "response": "Текст ответа модели...",
    "label": "bad"
  }'
```

Фидбек сохраняется в `data/bad_feedback.jsonl` для последующего анализа и улучшения модели.

## 🎓 Дообучение (LoRA)

### Формат данных

Обучающие данные хранятся в JSONL-формате. Каждая строка — JSON-объект:

```json
{"instruction": "Проблема со старым банковским терминалом", "output": "При проблемах со старым банковским терминалом необходимо..."}
```

Расширенный датасет также поддерживает поле `input` для дополнительного контекста:

```json
{"instruction": "Вопрос", "input": "Дополнительный контекст", "output": "Ответ"}
```

### Датасеты

| Файл | Записей | Описание |
|------|---------|----------|
| `data/train_data.jsonl` | 68 | Базовый набор: основные регламенты и инструкции |
| `data/train_data_expanded.jsonl` | 343 | Расширенный набор: дополнительные вариации формулировок |
| `data/bad_feedback.jsonl` | 3+ | Негативный фидбек от пользователей (растёт по мере эксплуатации) |

### Запуск дообучения

```bash
python scripts/finetune_lora.py
```

> ⚠️ **Важно:** перед запуском отредактируйте путь к датасету в `finetune_lora.py` (переменная `data_files`), указав актуальный абсолютный путь.

### Параметры тренировки

| Параметр | Значение | Пояснение |
|----------|----------|-----------|
| batch_size | 2 | На GPU |
| gradient_accumulation | 8 | Эффективный batch = 16 |
| epochs | 2 | Количество эпох |
| learning_rate | 2e-4 | Стартовая скорость обучения |
| warmup_ratio | 0.05 | 5% шагов на разогрев |
| scheduler | cosine | Косинусное убывание LR |
| precision | fp16 | Смешанная точность |

Обученные LoRA-адаптеры сохраняются в директорию `vikhr_nemo_12B_lora_out/`. При следующем запуске сервера они будут автоматически подхвачены модулем `model_loader.py`.

## 🔄 Цикл улучшения модели

Farmika спроектирована с учётом итеративного улучшения качества:

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  Пользователь│    │   Фидбек     │    │  Расширение  │    │  Дообучение  │
│  задаёт      │───▶│   "bad" →    │───▶│  датасета    │───▶│  LoRA        │
│  вопрос      │    │  bad_feedback │    │  (JSONL)     │    │  (finetune)  │
│              │    │  .jsonl      │    │              │    │              │
└──────────────┘    └──────────────┘    └──────────────┘    └──────┬───────┘
       ▲                                                          │
       │                      Улучшенная модель                   │
       └──────────────────────────────────────────────────────────┘
```

1. Пользователь получает ответ от модели
2. Если ответ некачественный — отправляет фидбек с меткой `"bad"`
3. Команда анализирует `bad_feedback.jsonl` и пополняет обучающий датасет корректными ответами
4. Запускает повторное дообучение LoRA-адаптеров
5. Обновлённые адаптеры подхватываются сервером

### Интеграция с Telegram-ботом

Farmika спроектирована как backend-сервис. Типичная схема интеграции — Telegram-бот, который пересылает вопросы сотрудников на `/ask` и возвращает ответы в чат:

```python
# Пример обработчика в Telegram-боте (aiogram)
@dp.message_handler()
async def handle_question(message: types.Message):
    async with aiohttp.ClientSession() as session:
        async with session.post(
            "http://localhost:8000/ask",
            json={"prompt": message.text}
        ) as resp:
            data = await resp.json()
            await message.reply(data["response"])
```

## 🐛 Известные ограничения

- **Галлюцинации** — несмотря на пост-обработку, модель может генерировать некорректную информацию, особенно по темам, не покрытым обучающим датасетом
- **Размер датасета** — 343 примера — это небольшой объём для fine-tuning; качество ответов напрямую зависит от расширения обучающей выборки
- **Жёстко заданные стоп-фразы** — список фраз для обрезки галлюцинаций задан в коде и требует ручного обновления
- **Путь к датасету** — в скрипте дообучения указан абсолютный путь Windows, который нужно менять под своё окружение
- **Отсутствие аутентификации** — API и админ-панель не защищены авторизацией
